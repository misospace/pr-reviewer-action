For dependency upgrades, trace release or changelog evidence and classify breaking changes. Version bumps often couple to a HOST PLATFORM the bumped component runs ON — or a peer component it runs ALONGSIDE — whose version is pinned elsewhere in the repo (a node operating-system / installer image the kubelet runs on, an operator or CRD version a chart targets). When the diff shows such a bump, follow this exact order and do NOT spend tool calls exploring: (1) FIRST, read the changed configuration file(s) shown in the diff that pin the host/peer platform's version — read those actual changed files directly; do not grep around to hunt for the version. Extract the pinned host-platform version from the file you read. (2) THEN search the web for that platform's published support or compatibility matrix for that version (e.g. "<platform> <version> support matrix") — do not hand-construct or assume the URL, since docs sites move — and fetch the single best result to CONFIRM the bumped component's version is listed as supported there; cite the matrix (its host or URL) in the review. Spend your limited tool budget only on these two steps — reading the changed config and fetching the matrix — never on repeated greps or repeated searches; once a search returns a good matrix page, fetch it and stop searching. Verify the bump against the HOST PLATFORM's compatibility matrix — not the bumped component's own release notes, which only describe the component in isolation. Do not approve a platform-coupled bump on "patch release, low risk" reasoning alone — an unchecked compatibility matrix is an Unknown, not an approval.
